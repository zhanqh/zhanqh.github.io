缘由

在实习公司做的广告管理后台的项目是跟一个正职的后端开发配合采用前后端分离的开发模式，跨域自然是一个不可避免的问题。项目一开始就遇到了，那时就直接让 leader 来解决，只知道大概的解决方案：用 nginx 做反向代理，将以 /api 开头的请求代理到后端的服务器，而其他的请求则全部代理至本地的 webpack 服务器。至于 nginx 服务器是怎么配置怎么使用那时是完全不懂的。后来在设置 Vue-router 开启 HTML5 的 history 模式时 nginx 服务器配置时向后端请教了一下，发现很有必要掌握基础的 nginx 配置。其实这是一个月前就开始在看的内容，结果却拖到了今天才写笔记。

简单讲 nginx 就是一个网页服务器，它能反向代理 HTTP, HTTPS, SMTP, POP3, IMAP 的协议链接，以及一个负载均衡器和一个 HTTP 缓存。对我来讲 nginx 目前最大的用途有两个，一是通过反向代理解决前后端分离开发带来的跨域问题，而是架设一个静态服务器进行本地开发、展示 demo 等。

反向代理

正向代理最常见的就是我们平时用来科学上网的 VPN ，多个终端连接到一台代理服务器上，代理服务器作为客户端的中介，再去请求目标服务器。

反向代理则是反过来，代理服务器作为目标服务器的中介，隐藏掉真正提供服务的服务器。

静态服务器

前端本地开发在没有使用 webpack 等自动化工具时，使用 nginx 搭建一个静态服务器是非常方便有效的。下周我要做的改版项目就是使用 nginx 架设本地服务器开发的。

项目前端部分使用 webpack 构建，webpack 监听本地8090端口，后端服务器在同一局域网内的 192.168.1.205 的 8888 端口上，通过 nginx 的反向代理，将项目中包含/api前缀的请求转发到后端服务器上，除此之外的全部请求都直接访问本地的 8090 端口。跨域问题就完美解决了，不用修改任何项目中的代码，无损的前后端分离开发。

暑假的时候我已经实现了实时公交查询，但是我没有解决跨域问题，因为我用 Express 框架搭建了服务器，使用 Nodejs 的中间件处理数据，转发 http 请求。具体过程就是，web 应用向服务器发起 ajax 请求，服务器收到请求后，通过中间件处理数据然后再向实时公交服务器发起 http 请求，等到实时公交服务器返回数据后 Nodejs 服务器再将数据返回给前端。毫无疑问这种处理的方式的性能是很落后的，时延比较长。好处就是我将密钥、接口参数这些重要信息放到了服务器，避免了借口参数被人知道，因为前端是无法加密的。而我自己的服务器我能够控制，随时能修改。

后记

在本地测试开发的时候，无意中我在 servers 目录内设置了两个配置文件，分别为wqn与zqh

wqn

而在 hosts 文件又设置了将这两个域名都解析到 127.0.0.1

这意味着有两个 nginx 虚拟主机监听到 127.0.0.1:80 上，但是事实上并没有冲突，访问 zqh.com 与 wqn.com 都符合预期，似乎没有冲突。而直接访问 127.0.0.1:80 时，则跳到了 wqn.com 上，一开始以为是 hosts 设置会有先后顺序，后边会覆盖前边，但是将两个 hosts 配置互换位置，发现结果还是一样。于是觉得可能会是 nginx 配置加载有先后顺序，把 wqn 改成 zqn，配置文件zqh就排在了前边，sudo nginx -s reload重启 nginx 服务器，访问 127.0.0.1:80 果然跳到了 zqh.com 上。

nginx 服务器应该是一次加载配置文件，直到找匹配项就结束，匹配不到就跳转到默认设置。所以通过域名去访问才会出现似乎是没有冲突的结果，通过 IP 访问就只会加载第一匹配的配置文件，进入这个虚拟主机了。利用这个规则似乎可以在一个 IP 上监听多个相同端口号（例如默认的80），然后使用网址去访问。
